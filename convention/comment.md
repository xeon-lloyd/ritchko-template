# 주석 컨벤션

## 목적
- 이 문서는 이 레포에서 새로 쓰거나 수정하는 코드의 기본 주석 정책을 정의한다.
- 목표는 주석 수를 늘리는 것이 아니라, 적고 오래가며 실제로 도움이 되는 주석만 남기는 것이다.

## 핵심 원칙
- 좋은 코드는 이름과 구조만으로 무엇을 하는지 드러내야 한다.
- 주석은 코드만으로 복원하기 어려운 이유, 제약, 의도, 트레이드오프, 실수하기 쉬운 지점을 설명할 때만 쓴다.
- 주석이 필요해 보이면 먼저 이름, 상수, 함수 분리, 흐름 단순화로 해결 가능한지 본다.

## 기본 정책

### 1. 기본값은 적은 주석이다
- 기본적으로 주석을 추가하지 않는다.
- 이름을 더 명확하게 바꾸거나, helper를 분리하거나, 매직 넘버를 상수로 바꿔 주석 필요를 줄인다.
- 코드 구조를 개선해서 해결할 수 있으면 주석보다 코드 개선을 우선한다.

### 2. 아래 경우에만 주석을 추가한다
- 왜 이런 결정을 했는지 설명해야 할 때
- 더 단순해 보이는 대안을 왜 쓰지 않았는지 설명해야 할 때
- 중요한 제약, invariant, business rule이 숨어 있을 때
- 브라우저, 라이브러리, 외부 API, 플랫폼의 quirks를 우회할 때
- 성능, 보안, 동시성, 정합성과 관련된 함정을 막아야 할 때
- 미래 유지보수자가 쉽게 깨뜨릴 수 있는 가정을 보호해야 할 때
- 수식, 순서, threshold, edge case 처리가 코드만으로는 납득되지 않을 때

### 3. obvious한 코드 번역 주석은 금지한다
- 코드 한 줄을 한국어 문장으로 다시 풀어쓴 주석은 쓰지 않는다.
- `return total` 위에 `// total 반환` 같은 주석은 금지한다.
- `if (!user) return ...` 위에 `// user가 없으면 return` 같은 주석도 금지한다.

### 4. 주석은 오래 살아남아야 한다
- 구현 세부를 줄줄이 설명하는 주석은 쉽게 낡으므로 피한다.
- 변경 이력, 작성자, 날짜, 예전 구현, 주석 처리된 코드는 남기지 않는다.
- Git이 대신할 수 있는 정보는 코드에 남기지 않는다.

### 5. 주석은 구체적이어야 한다
- `// 나중에 수정`
- `// 임시`
- `// thread safe 아님`
- 같은 모호한 주석은 금지한다.
- TODO는 제거 조건이나 후속 행동이 분명할 때만 남긴다.

권장:

```js
// TODO: 벤더 SDK의 재시도 중복 호출 문제가 해결되면 이 우회 로직 제거
```

### 6. 주석 언어
- 기본적으로 한국어로 쓴다.
- 이미 팀 내에서 굳은 영어 용어는 그대로 둔다. 예: `token`, `API`, `stream`, `cache`
- 문체는 짧고 사실 중심으로 쓴다.

### 7. 주석 배치
- 주석은 설명할 코드 바로 위에 둔다.
- end-of-line 주석은 짧은 보충 설명이 정말 필요할 때만 쓴다.
- 짧은 일반 주석은 `//`를 사용한다.
- 외부 소비자가 읽는 공개 계약 문서는 필요할 때만 doc comment를 사용한다.

## 이 레포의 예외 없는 기본값
- public API, 공용 utility, 외부 계약은 필요한 경우 문서 주석으로 입력/출력/제약을 설명한다.
- 내부 구현 함수는 주석보다 이름과 구조로 설명하는 것을 우선한다.
- 수정 중인 코드에 오래됐거나 오해를 부르는 주석이 보이면 같이 정리한다.

## backend 예외
- backend operation 로직 파일과 `module/` 파일은 흐름 파악이 중요하므로 짧은 한국어 단계 주석을 유지한다.
- 아래 단계는 필요 시 우선적으로 주석을 붙인다.
  - `// 입력값 검증`
  - `// 대상 존재 여부 확인`
  - `// 권한 확인`
  - `// 핵심 비즈니스 처리`
  - `// 응답 반환`
- 이 예외는 "주석을 많이 써라"가 아니라 "큰 흐름이 한눈에 보이게 유지하라"는 의미다.
- 여전히 단순 대입, 단순 return, obvious한 if문마다 주석을 달면 안 된다.

예시:

```js
module.exports = async function(param, req, res){
    // 입력값 검증
    if(!param.signalId) return new response.FormInputRequired('signalId');

    // 대상 존재 여부 확인
    const signal = await getSignalById(param.signalId);
    if(!signal) return new response.SignalNotFound();

    // 핵심 비즈니스 처리
    await updateSignalStatus(signal.id, 'archived');

    // 응답 반환
    return new response.ArchiveSignalOK();
}
```

## 제거 대상 주석
- obvious한 주석
- 이름만 제대로 바꾸면 사라질 주석
- stale할 가능성이 높은 주석
- 원인과 영향이 없는 막연한 경고
- 소유자나 조건이 없는 TODO
- 변경 이력, 작성자, 날짜
- 주석 처리된 구 구현

## 최종 점검
- 더 좋은 이름이나 함수 분리로 이 주석을 없앨 수 없는가
- 이 주석이 무엇이 아니라 왜를 설명하는가
- 리팩터링 후에도 참일 가능성이 높은가
- 미래 유지보수자의 실수를 실제로 막아주는가
- backend가 아니라면 이 주석이 정말 필요한가

한 줄 요약:

**obvious한 내용은 코드가, non-obvious한 이유와 제약은 주석이 담당한다. backend는 그 원칙 위에서 짧은 한국어 단계 주석을 유지한다.**
